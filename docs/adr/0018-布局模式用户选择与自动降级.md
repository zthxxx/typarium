# ADR 0018 —— 布局模式用户选择与自动降级

- 状态：已采纳
- 日期：2026-07-22

## 背景 Context

[ADR-0017](./0017-双布局引擎-euler-与-hasse-自动切换.md) 引入了 Euler / Hasse 双引擎，切换完全自动：`computeCanvasLayout` 在 core 内先跑 faithfulness 探测，零违例走 Euler，否则落 Hasse。用户无法主动选择图型；产品上需要显式的「图类型」radio 选择器，且语义分层：用户没选过默认 Euler；Euler 画不出时自动禁用该选项并展示 Hasse；一旦代码恢复可画且用户未主动选过 Hasse，自动回到 Euler；只有用户主动选择 Hasse 后才固定不再自动恢复

## 决策 Decision

1. 切换策略从 core 上移到 `VisualizationStore`：core 只保留三个纯函数（`probeRectFaithfulness` / `computeRectLayout` / `computeHasseLayout`），`computeCanvasLayout` 的自动切换封装移除，`CanvasLayout` discriminated union 类型保留
2. 状态模型：`userChoice: 'euler' | 'hasse' | null`（null = 从未选过，不持久化，会话内有效）；`eulerDrawable` = probe 零违例（computed）；`effectiveMode` = userChoice 为 hasse 时 hasse，否则 eulerDrawable ? euler : hasse（computed）
3. UI 上 Euler 选项在不可画时禁用并显示 Hasse 选中；选择器带 info 说明 Popover，示例图用真实布局引擎渲染固定输入，不用截图资产

## 代价与后果 Consequences

- core 少了一个「开箱即用」的自动切换入口，嵌入方需要自行组合 probe 与两个引擎；换来的是策略（用户意图）与机制（纯函数布局）分层清晰，两个引擎互不知晓、可独立拆包
- userChoice 不持久化：跨会话回到默认 Euler。图型偏好与具体代码内容强相关，继承旧选择容易造成「为什么默认不是 Euler」的困惑；将来要持久化只需一行 localStorage

## 备选方案 Alternatives

- 在 core 的 `computeCanvasLayout` 上加 force 参数 —— 否决：用户意图是应用状态，塞进纯函数层会让 core 承载 UI 语义，拆包后每个嵌入方都被迫理解该参数的产品含义
- userChoice 持久化到 localStorage —— 否决（当前期）：见代价一节
