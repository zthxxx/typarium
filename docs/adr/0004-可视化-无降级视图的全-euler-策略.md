# ADR-0004 可视化：无降级视图的全 Euler 策略

- 状态：已采纳（推翻此前调研建议）
- 日期：2026-07-16

## 背景 Context

前期调研曾建议：Euler 图不可读时降级为 linear diagram（实证任务表现优于 Euler，[TOCHI 2015](https://dl.acm.org/doi/10.1145/2810012)）或 Hasse 图。需求方否决了降级视图，理由：所有类型都能用集合论表达，那么所有类型就都应该能用 Euler 图表达 —— 产品的教学主张本身就是「类型即集合」，切到非集合视图等于自己拆台

## 决策 Decision

1. **只有 Euler 一种视图**。空画布也是 Euler 图（unknown 全集 + never 点阵）；任何合法输入都在 Euler 范式内呈现
2. 唯一例外是 `any`：它不属于集合论体系，用悬浮 badge 移出图外（[ADR-0005](./0005-never-点阵与-any-悬浮-badge.md)）
3. 期 1 范围（原始类型 + 结构体 + 函数 + union/intersection）的可绘性由固定底图按构造保证（[ADR-0003](./0003-语义引擎-固定基底域与动态原子.md)）
4. 复杂对象格局判定不了的相交性，用「重叠未知」虚线态呈现 —— 它仍是 Euler 图内的表达，只是边界样式诚实
5. 导出数量超上限时 toast 提示删减导出，视图形态不变

## 代价与后果 Consequences

- 极端格局（大量同域对象两两部分重叠）下图会拥挤、可读性下降，用户只能靠删减导出自救 —— 接受，因为教学场景的导出数量天然少
- 「重叠未知」承担了原本降级视图的职责，它的视觉语言（虚线 + hover 解释）必须在渲染层实现到位，否则等于静默说谎
- 调研中 linear diagram 的可读性优势被放弃 —— 教学一致性优先于任务效率，这是产品判断

## 备选方案 Alternatives

- linear diagram / Hasse 降级：任务表现更好但脱离集合隐喻，与产品主张冲突，被需求方否决
- UpSet 图：面向基数分析，且 UpSet.js 主库 AGPL + 商业双许可，license 不可引入，否决（附注：本仓库不引入 GPL/AGPL 依赖）
