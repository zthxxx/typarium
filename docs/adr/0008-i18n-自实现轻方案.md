# ADR-0008 i18n 自实现轻方案

- 状态：已采纳
- 日期：2026-07-16

## 背景 Context

需求是中文 / English 两种界面语言：默认读浏览器语言、用户切换后持久化。UI 文案量级小（工具页面，非内容站点）

## 决策 Decision

自实现：`Record<MessageKey, string>` 两份字典（`i18n/zh.ts`、`i18n/en.ts`，key 用 TS 联合类型约束全覆盖）+ `SettingsService` 的 mobx observable locale + `t(key, params?)` 插值函数。默认 `navigator.language`（`zh` 前缀 → 中文），切换写 IndexedDB

## 代价与后果 Consequences

- 没有复数规则、日期数字格式化、命名空间懒加载 —— 本产品文案没有这些需求；插值用模板函数覆盖
- 将来若语言数量增长或需要复杂格式化，迁移到 i18next 一类框架的成本 = 字典格式转换 + `t` 调用点不变（保持同签名），可控
- key 全覆盖由类型系统保证（两份字典共用同一个 key 联合类型），漏译直接编译报错 —— 这是框架方案反而给不到的强约束

## 备选方案 Alternatives

- react-i18next / lingui：为 2 个 locale 引入运行时 + 构建插件 + 学习成本，收益为零，否决
- 浏览器 `Intl` 上再包框架：本产品没有格式化需求，否决
