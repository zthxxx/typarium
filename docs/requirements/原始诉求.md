# 原始诉求

> 本文按时间记录需求：第一轮为初始形态，[第二轮需求（2026-07-16）](#第二轮需求2026-07-16)对可视化范式、分析引擎、预设区与页面布局做了大幅修订 —— 两轮冲突处以第二轮为准，冲突清单见该章节开头

typarium 用一句话定义：一个教学游戏化的 Web 页面，用户输入 TypeScript 类型定义，页面实时把这些类型之间的集合关系画成 Euler 图 —— 用「类型即值的集合」这一心智模型讲清 TypeScript 的代数数据类型（ADT）

做它的原因：「类型即集合」在社区已是公认的正确心智模型（[TS 官方 handbook](https://www.typescriptlang.org/docs/handbook/typescript-in-5-minutes-oop.html)、[Effective TypeScript Item 7](https://github.com/danvk/effective-typescript/blob/main/samples/ch-types/types-as-sets.md)、[thoughtspile 的集合论讲解](https://thoughtspile.github.io/2023/01/23/typescript-sets/) 都在讲），但所有讲解停留在静态手绘配图，「自定义类型 → 实时 Euler 图」的交互工具是空白。这个工具直接服务于作者的分享演示：讲协变/逆变/双向协变、Tagged Union 时，在代码框里现场敲类型、图上立刻看到集合关系的变化

## 可视化语义要求

- `unknown` 是全集：铺满整个可视化画布；画布上存在任何其他类型时，`unknown` 都包含它们
- `never` 是空集：以「点阵」画法呈现 —— 类比 Photoshop 里透明背景显示为浅色棋盘格，`never` 的点阵铺在所有集合区域内，表达「空集被除自身以外的所有集合包含、无处不在」；每个点用 `∅` 字形（或等价图形）绘制，需要考虑渲染性能
- `any` 不属于集合论体系：以一个可拖拽的悬浮 badge 呈现（类似客服助手悬浮球），默认位置横跨可视化画布与代码编辑区的边界中间偏上，斜置约 15°，橙红色，带投影阴影 —— 投影表达它不在 Euler 图的层级里，图内其他集合一律无投影；hover 后提示 any 不属于集合范畴、会打破集合约束
- 字面量类型是单元素集：`"foo"` 画成 `string` 区域内的一个小环
- `{}` 是「全集减去 `null` 和 `undefined`」：一个跨越多个基底域的大轮廓，本身就是教学案例（`string` 也是合法的 `{}`，见 [TS FAQ](https://github.com/microsoft/TypeScript/wiki/FAQ#primitives-are---and---doesnt-mean-object)）
- union 类型画成包住各成员区域、中间收腰的轮廓；包含关系用嵌套轮廓表达，嵌套之间留 padding
- 相互可赋值的类型合并为同一区域、共用多标签
- 引擎无法判定相交性的类型对，用虚线边界的「重叠未知」状态诚实呈现，语义与判定边界见[功能技术设计](../design/功能技术设计.md)

## 布局与 UI 要求

- 整体布局仿 [TS Playground](https://www.typescriptlang.org/play/)：Header/Footer 承载站点信息、设置与跳转；内容区左侧为集合可视化画布、右侧为代码编辑器
- Header 有语言选择器（仿 [astexplorer](https://astexplorer.net/) 的交互位），本期仅 TypeScript 可选且默认选中，架构上保留多语言扩展点
- UI 风格是教学游戏：配色清新、边框偏粗线条、整体灵动，有科技感，色系贴近 TypeScript/JavaScript 品牌（蓝 / 黄 / 橙 / 白）
- 集合区域：边框粗且不透明；背景色淡、半透明、带格子或点阵花纹（非纯色）；不同集合用不同的边框/背景色区分
- 边框有轻微的流动、波动变化，像在呼吸（参考 Patrick's Parabox 的边缘效果），让页面有「活着」的感觉
- 可视化画布不可缩放：只有最小尺寸约束（适配手机端时保持最小尺寸，页面滚动）
- 先确定 design token（色板、边框、圆角、动效时长），再做界面

## 能力要求

- 只把 `export` 的类型作为结果展示；带类型参数且参数没有完整默认值的泛型不展示（全默认参数的泛型可裸引用，视为默认实例化）。示例：

  ```ts
  export type UnionBoolean<T> = T | boolean

  export type R1 = UnionBoolean<string>
  export type R2 = UnionBoolean<number>

  type R3 = UnionBoolean<unknown>
  ```

  上例只展示 `R1` 与 `R2`：`UnionBoolean<T>` 因泛型参数无默认值不展示，`R3` 因未 export 不展示

- 编辑器接入 monaco：输入后自动做语法与类型校验，报错在编辑器内对应位置高亮提示，形式等同 VS Code；TypeScript strict check 常开
- 校验通过后 1.2s 内无新输入，自动提交可视化
- hover 到画布上集合名字时，仿 VS Code LSP 展开类型推断结果；编辑器内 hover 同样展示类型推断
- 可视化画布顶部固定一排预设类型按钮：`any` `unknown` `object` `void` `undefined` `null` `never` `{}`（取自 [TS handbook 的可赋值性矩阵](https://www.typescriptlang.org/docs/handbook/type-compatibility.html#any-unknown-object-void-undefined-null-and-never-assignability)），点击后自动加入展示，无需手写代码
- i18n 支持中文 / English：默认读浏览器语言，用户切换后持久化保存
- 所有用户输入本地保存（IndexedDB），刷新或重新进入页面后恢复
- 分享：可选择把当前内容一起编码进链接（URL-safe Base64 变体编码进 hash），他人打开即恢复同样效果；`Cmd / Ctrl + S` 把分享链接写入剪贴板并提示

## 期 1 范围

期 1 至少支持以下类型形态的输入与可视化，因为它们是教学演示的素材：

1. 所有原始类型（含 `any`）
2. 结构体（对象）类型
3. 函数类型
4. 以及它们的 union / intersection

服务的教学主题：协变与逆变、方法参数的双向协变、Tagged Union（Sum Type）。验收标准是下方「教学演示用例」全部可以在代码框输入并正确呈现

## 教学演示用例

以下用例来自作者的分享笔记，是期 1 的验收场景。每个用例给出输入代码与预期的集合关系

### 协变：类型参数方向与结果方向一致

```ts
type CoVariance<T> = T | boolean

export type CoNarrow = CoVariance<string> // string | boolean
export type CoWide = CoVariance<string | number> // string | number | boolean
```

预期：`CoNarrow ⊂ CoWide`，与 `string ⊂ string | number` 方向一致 —— 输入更小，得到更小

### 逆变：条件类型反转包含方向

```ts
type ContraVariance<T> = [T] extends [string] ? boolean | number : boolean

export type ContraFromNarrow = ContraVariance<string> // boolean | number
export type ContraFromWide = ContraVariance<string | number> // boolean
```

预期：`ContraFromWide ⊂ ContraFromNarrow` —— 输入更大，得到更小，包含方向翻转

### 函数参数是逆变位置

```ts
type Fn<X> = (params: X) => void

export type FnNarrowParam = Fn<string>
export type FnWideParam = Fn<string | number>
```

预期：`FnWideParam ⊂ FnNarrowParam` —— 参数范围更大的函数是「更精确」的子集（strict 模式下函数参数按逆变检查）

### 方法参数双向协变 vs 属性函数逆变

```ts
interface Animal {
  name: string
}
interface Dog extends Animal {
  breed: string
}

// method shorthand: params stay bivariant even under strictFunctionTypes
export interface Kennel {
  addAnimal(animal: Animal): void
}
export interface DogKennel {
  addAnimal(dog: Dog): void
}

// property function: params checked contravariantly
export interface StrictKennel {
  addAnimal: (animal: Animal) => void
}
export interface StrictDogKennel {
  addAnimal: (dog: Dog) => void
}
```

预期：`Kennel` 与 `DogKennel` 相互可赋值 → 合并为同一区域多标签，并带「双向协变」偏差标注（这是 TS 故意保留的不健全角落）；`StrictKennel ⊂ StrictDogKennel` 呈现严格包含 —— 两组并排看，双向协变的教学冲击力最大

### Union to Intersection

```ts
type UnionToIntersection<U> = (U extends any ? (x: U) => void : never) extends (
  x: infer I,
) => void
  ? I
  : never

export type U = { a: string } | { b: number }
export type I = UnionToIntersection<U> // { a: string } & { b: number }
```

预期：`I ⊂ { a: string } ⊂ U` 且 `I ⊂ { b: number } ⊂ U`，交集嵌在两个成员的重叠处

### Tagged Union（Sum Type）vs 宽松可选字段

```ts
export type GroupRow = { kind: 'Group'; groupName: string }
export type DataRow = { kind: 'DataRow'; data: Record<string, string> }
export type CreatorRow = { kind: 'Creator'; authorID: number }

export type RowData = GroupRow | DataRow | CreatorRow

export type LooseRowData = {
  kind: 'Group' | 'DataRow' | 'Creator'
  groupName?: string
  data?: Record<string, string>
  authorID?: number
}
```

预期：三个 variant 两两不相交（判别字面量冲突，引擎可判定）；`RowData` 是包住三者的收腰轮廓；`RowData ⊂ LooseRowData` 且是真包含 —— 图上直接看到宽松写法多出来的非法组合空间，这就是 Sum Type 的价值

### `{}` 的语义

```ts
export type Anything = {}
export type Str = string
```

预期：`{}` 的轮廓覆盖除 `null` / `undefined` 外的一切，`string ⊂ {}` 清晰可见

## 架构约束

- 底层设计为多语言 ADT 类型集合推断的扩展架构（语言 adapter），本期只实现 TypeScript，但交互 UI 与代码架构保留扩展点
- 逻辑不写在 View 层：UI 只承载「数据 → 渲染」与「UI 事件 → 领域方法」的转换，逻辑在 service 中；power-di 做依赖注入、service 建模；React 数据流用 mobx
- 类型解析等计算放在 Web Worker 中执行
- TypeScript strict check 常开

## 工程约束

- 技术栈：React + TanStack Start，Vite 构建（voidzero 生态）
- 一个合格的框架仓库需要：单元测试、端到端测试、Code Lint、Code Format、Commit Hook；仓库出现素材时启用 Git LFS
- 代码与注释一律工程化英文；docs/ 是唯一中文区
- 部署：GitHub Pages，CNAME 为 `typarium.zthxxx.me`（域名解析由持有者在验收后配置，此前一律本地浏览器验证）
- 仓库启用 CI（GitHub Actions）
- 开发节奏：完成一个步骤并通过验证后即提交一个 git commit；有争议的决策先写 [ADR](../adr/) 再实现

## 第二轮需求（2026-07-16）

以下为需求方在第一版浏览器验收后给出的修订与新增。与第一轮冲突并被取代的条目：

- 可视化从 bubblesets 不规则轮廓 + 固定基底域底图，改为**圆角矩形包含布局 + 空画布**（取代「可视化语义要求」中 union 收腰轮廓、基底域相关描述）
- 类型分析引擎从 TS 5.9 checker 改为 **TypeScript 7 tsgo-wasm**
- 预设区从 8 按钮插代码，改为 **26 项双轨目录**（取代「能力要求」中预设按钮一节）
- 窄屏从上下堆叠改为**编辑器浮出抽屉**；Footer 移除 GitHub 链接

### 响应式与页面布局

- 页面及画布整体尺寸（含画布与标题、编辑器的位置关系）根据屏幕宽度自适应全屏，支持窄屏简易适配
- 窄屏时编辑器从右侧浮出、盖在屏幕上；点击折叠直接收回去，明确否决「放到画布底下、向下滚动查看」的形态
- 代码编辑区域可拖拽宽度、可收起
- 画布没有任何类型时显示为空，不在背景显示默认的灰色类型集合底图
- 不显示 GitHub 右下角链接

### 分析引擎

- 类型分析引擎使用 TypeScript 7 [tsgo-wasm](https://www.npmjs.com/package/tsgo-wasm)

### 预设可选项

按分类排列展示（顺序即分类顺序，不渲染分组样式）：

- 基本类型（8）：`string` `number` `boolean` `bigint` `symbol` `null` `undefined` `object`
- 内置类型（4）：`unknown` `never` `void` `any`
- 常见类型（3）：`Array<T>` `Object` `Function`
- 预设代码类型（11）：`string & number`、`string | number`、`string | number | boolean`、`() => string`、`() => number`、`() => string | number`、`(_: string) => void`、`(_: number) => void`、`(_: string | number) => void`、`<T extends string>(_: T) => void`、`<T extends string | number>(_: T) => void`

行为约定：

- 非代码类预设（基本 + 内置 + 常见）点击后编辑器代码零变化，直接在画布展示；这些项必须直接可见，不允许收进浮层；首选区域可以多行
- 展示不下的部分（代码模板类）hover / 点击展开浮层选择；默认至少直接显示基本类型与内置类型
- `any` 任何时候都要有特殊警示样式（淡红色调），提示它的特殊性
- 代码类预设点击后向编辑器插入 `export type C1 = ...`、`export type C2 = ...` 递增编号的声明行；若插入行与文件最后一行之间没有独立空行则补一行，连续操作形如：

  ```ts
  export type C1 = ...

  export type C2 = ...

  export type C3 = ...
  ```

### 矩形包含布局（画布）

不再用不规则包围图形表达包含关系，仅通过圆角矩形的位置编排体现包含或不包含 —— 本产品的表达模型里 TypeScript 类型只有包含与不包含，不引入「部分有交集」的概念。图形比例与面积无语义，直接平分：

- 只有一个元素时占满整个画布
- 多个互不包含的元素平分画布：两个对半、三个平分
- `A ⊃ B` 时：B 占 A 的一半，剩下一半表示 A 内的其他类型（other 区域面积与数量无关，只是平分中的一份）
- `A ⊃ {B, C}` 且 B、C 互不包含时：A 占满画布，B、C 各占约三分之一，因为 A 除 B、C 外还有其他类型
- 元素个数不满偶数时向上取整到偶数计算布局，对该偶数做双因子分解，横竖个数尽可能均分且横 ≥ 竖
- 两个元素完全等价（互为父子）时占据完全相同的区域；嵌套显示为多一层边框，层间有一点 padding
- 点击 never 时出现整体背景点阵；hover 到非类型区域的空白（如圆角边缘）时点阵上浮出 popover 提示它是 never
- 多个类型重叠处的 tooltip 用多项列表表达（逐层列出「名字: 类型文本」）
- 参考图仅表达原型效果；实际样式要求更有设计感、游戏感（UI 实现时加载 design-taste-frontend 技能）

参考图描述（三张原型示意）：其一，两组 string 集合并列、内部各有字面量小矩形，背景为点阵；其二，unknown 为最外层矩形，内含 string（含 'a1' 'a2' 字面量）与 number（含 1、2 字面量）两个并列矩形，矩形外的 unknown 内部区域为点阵；其三，hover 点阵处浮出列表式 tooltip，逐行列出 `A2: 'a1' | 'a2' | 'a3'` 与 `unknown: ...`
